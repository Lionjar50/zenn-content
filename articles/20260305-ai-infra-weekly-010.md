## 1. Cloudflare Security Week 2026 — WAF・Zero Trust・脅威レポートの三本柱

**🔗 出典: [Cloudflare Blog](https://blog.cloudflare.com/attack-signature-detection/)**

Cloudflareが「Security Week 2026」として複数の重要発表をまとめて行いました。

### Attack Signature Detection（WAF刷新）

従来のWAFは「ログモードで様子見するか、ブロックモードで誤検知リスクを取るか」という二択を迫られてきました。今回発表の **Attack Signature Detection** と **Full-Transaction Detection** は、リクエストのペイロードとサーバーレスポンスを相互に突き合わせることで、**実際に攻撃が成功したかどうか**を常時・高精度で判定します。

これにより「ブロックして様子見→チューニング→また誤検知」というループから解放され、継続的な保護と低False Positiveを両立できます。

### ポスト量子暗号をSASEに統合

CloudflareはOne（SASE）プラットフォームの全コンポーネント——Zero Trust Access、Secure Web Gateway、WAN-as-a-Service——にポスト量子暗号（PQC）対応を展開しました。量子コンピュータによる"Store Now, Decrypt Later"攻撃への備えが、エンタープライズネットワーク全体で現実的な選択肢になりました。

### 2026 Cloudflare Threat Report

初の年次脅威レポートで最も衝撃的だったのは「**サイバー脅威の完全な産業化**」という表現です。Cloudflareが1日平均で**2,300億件**の脅威をブロックしているというスケール感は、攻撃が手作業からAIドリブンな自動化へ移行していることを裏付けています。

**実務への影響:** WAFを運用しているチームは、Attack Signature Detectionの早期アクセスを申し込む価値あり。PQCについては「いつかやる」から「今期の検討事項」へ格上げするタイミングです。

**個人的な感想:** WAFのチューニング作業って本当に不毛で、「誤検知を減らしたい」→「ルールを緩める」→「今度は抜けが増える」の無限ループになりがちです。Attack Signature Detectionは「リクエストを見るだけでなくレスポンスと突き合わせる」という発想が根本的に違う。2,300億件/日という数字は正直ピンとこないですが、もはやCloudflareが人類のインターネット防衛インフラになってしまっているという事実は、少し複雑な気持ちになります（依存度的な意味で）。

## 2. Google Cloud × MWC 2026 — AIエージェントが通信網を自律管理

**🔗 出典: [Google Cloud Blog](https://cloud.google.com/blog/topics/telecommunications/autonomous-networks-at-mwc-2026)**

Barcelona開催のMWC 2026で、Google CloudはAIエージェントを使った**自律ネットワーク運用フレームワーク**を発表しました。

### 何が変わるのか

従来の「AIによるネットワーク監視」から一歩進み、今回のAIエージェントは**実際にアクションを執行**します。たとえば：

- 音声コアネットワークの輻輳を検知→自律的にトラフィックを迂回
- 障害発生時にネットワーク設定を自動リセットして通話品質を回復

One NZがすでにトライアル中で、「監視する人間が常にいなくてよい（Zero Touch）」Level 4〜5の自律性を目指しています。

また、DigitalRouteとの連携で**AI-Readyなデータパイプライン**も発表。生のネットワークログを構造化してAIエージェントが学習・判断しやすい形に変換する仕組みです。

**実務への影響:** これはテレコムの話に見えて、実は**データセンター内ネットワーク自動化の方向性**と地続きです。BGPやVLAN管理のAIエージェント化を検討する際の参考アーキテクチャとして要注目です。

**個人的な感想:** ネットワークエンジニアとしての正直な感想を言うと、「AIが自律的にBGPを操作する」というのはまだ怖い。障害対応で予期しないルーティング変更が入ったらトラブルシューティングが地獄になる。One NZのトライアルがどの程度の権限を与えているのかが気になるところです。とはいえ、「人間が設定変更のたびにticketを切る」現状も限界なのは事実で、Level 3（人間の承認が必要）くらいから慣らしていくのが現実的かなと思います。

## 3. GGML / llama.cpp が Hugging Face に合流 — ローカルAI基盤の長期保護

**🔗 出典: [Hugging Face Blog](https://huggingface.co/blog/ggml-joins-hf)**

llama.cppの生みの親 Georgi Gerganov 氏率いる **ggml.ai チームがHugging Faceに合流**しました（2月下旬発表）。

### なぜ重要か

llama.cppはローカルLLM推論の事実上の標準ライブラリです。企業のオンプレLLM運用、Raspberry Pi上のエッジAI、コード補完ツールまで、ありとあらゆる「クラウドなしAI」を支えています。

合流の目的は明確で、「**AIをオープンかつローカルに保つための長期的な制度基盤を確立する**」こと。プロジェクトはオープンソースのままで、コミュニティの自律性も維持されます。

**インフラ観点のポイント:**
- gguf フォーマットの標準化・長期サポートが期待できる
- HFのモデルハブとの統合深化→オンプレデプロイのパイプラインが簡素化
- 企業がローカルLLMを「将来的に消える OSS」として懸念するリスクが低下

**個人的な感想:** 今週一番「よかった」と思ったニュースです。llama.cppは個人開発者が作った「趣味プロジェクト」から始まり、今やローカルLLMの根幹を支えるライブラリになっています。こういうプロジェクトは突然メンテナが燃え尽きてアーカイブになるリスクが常にある。HFへの合流は技術的な話より「持続性の担保」として意義が大きい。自分でOllamaを使ったローカルLLM環境を組んでいると、この手の基盤の安定性がいかに大事かが身に染みてわかります。ggufフォーマットが事実上の標準として固まってくれると、モデルの乗り換えコストが下がるのも嬉しい。

## 4. .NET が io_uring 統合で CPUを最大40%削減

**🔗 出典: [Qiita 解説記事](https://qiita.com/ishisaka/items/9d3c73fbca29d287085e)**

Microsoftの .NETが、Linuxカーネルの **io_uring** をソケット層に本格統合しました。

### 技術的な背景

従来の `epoll` はシステムコールのたびにカーネル↔ユーザー空間を往復し、Spectreなどのセキュリティパッチでさらにコストが増加していました。io_uringは**共有メモリ経由の完了駆動型モデル**で、コンテキストスイッチを最小化します。

.NETの実装では：

```
SQPOLL モード → システムコール・ゼロで動作可能
→ CPU使用量 15〜40% 削減（高スループットワークロードで）
→ ネイティブライブラリ不要（C#から直接カーネルメモリ操作）
```

**実務への影響:** LinuxホストでASP.NETやgRPCサービスを動かしている場合、ランタイム更新だけでCPU課金がそのまま削減されます。クラウドのvCPUコスト最適化を検討しているなら、カーネルバージョンとio_uring対応状況の確認を。

**個人的な感想:** io_uring自体は2019年頃から注目されていたので「ついに.NETまで来たか」という感じです。Linuxカーネルの非同期I/Oインターフェイスがこれだけ時間をかけて各ランタイムに普及していくのを見ると、低レイヤーの技術が実際の現場に届くまでのリードタイムの長さを実感します。「最大40%削減」という数字はベンチマーク上の話で、実際のワークロードでは10〜20%が現実的だと思いますが、それでもコード変更ゼロで出るならランタイム更新する理由としては十分です。

## 5. 国内トピック — Claude Opus 4.6 / Gemini 3 対応、AI Agent プラットフォーム更新

**🔗 出典: [Web担当者Forum](https://webtan.impress.co.jp/n/2026/03/04/52252)**

ギブリーの「MANA Studio」が最新LLMに対応し、**Claude Opus 4.6 / Sonnet 4.6（Reasoning対応版含む）** と **Gemini 3 Flash / 3.1 Pro** をサポートしました。

エンタープライズ向けAIエージェントプラットフォームが最新モデルへ追従するスピードが上がっており、「モデル選定→本番反映」のリードタイムが月単位から週単位へ短縮されつつあることを示しています。

**個人的な感想:** モデルの追従スピードが週単位になってきたのは、ユーザーにとっては嬉しいことですが、運用側は大変です。モデルが変わるたびにプロンプトの挙動が微妙に変わるので、回帰テストを自動化していないと品質の担保が難しい。「Claudeが変わったらRAGの精度が下がった」みたいなことが普通に起きる時代になっています。個人でAIエージェントを運用していると特に感じますが、モデルを固定できる仕組みの重要性が上がっていると思います。

## 🛠 今週のTips: Cloudflare WAF のチューニングを減らす設計

今週のCloudflare発表に絡んで、WAF運用を楽にする設計の考え方をまとめます。

**① マネージドルールセットを「ログモード」で展開して学習期間を設ける（2週間目安）**

```bash
# Terraform例: WAFルールをログのみで開始
resource "cloudflare_ruleset" "waf" {
  kind    = "zone"
  phase   = "http_request_firewall_managed"
  rules {
    action = "log"           # まず観察
    # action = "block"       # チューニング後に切り替え
    ...
  }
}
```

**② False Positive が多いパスはスキップルールで除外**

```bash
# /api/v1/upload のような大きなPOSTは先にスキップ
expression = (http.request.uri.path matches "^/api/v1/upload")
action     = "skip"
```

**③ Attack Signature Detection が展開されたら、ログ→ブロックの移行判断を「リクエスト単位」ではなく「成功した攻撃件数」で行う**

従来は「誤検知率X%以下になったらブロック」という基準でしたが、新機能では「実際に通ってしまった攻撃X件」という直感的な指標で判断できます。
