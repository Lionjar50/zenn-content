## はじめに

2025年末から2026年にかけて、コーディングエージェントの「2強」が出揃った。

- **Claude Code**（Anthropic、2025年2月リリース）
- **Google Antigravity**（Google、2025年11月リリース）

アプリ開発者向けの比較記事は増えてきたが、**インフラ・運用系エンジニア視点**の記事はまだ少ない。

Terraform、Ansible、シェルスクリプト、監視設定、CI/CDパイプライン——こういった作業を日常的に扱うエンジニアにとって、どちらのツールが実用的なのかを比較する。

---

## それぞれの設計思想

### Claude Code — 「ターミナルに住む」エージェント

Claude Codeは**CLIファースト**のツールだ。既存のターミナル環境に入り込み、ファイル操作・コマンド実行・コード編集を自律的にこなす。

```bash
claude "このTerraformのplan結果を見てエラーの原因を特定して修正して"
```

特徴：
- **エディタ非依存**。vimでもVS Codeでも、SSHセッションでも動く
- **ローカルファイルへの直接アクセス**。read/write/editをコードベース全体に対して実行できる
- **既存ワークフローへの統合がしやすい**。CI/CDのスクリプト内から呼び出すことも可能
- 料金：**Claude Pro（$20/月）のOAuth認証**で使える。APIキーによる従量課金も選択可

### Google Antigravity — 「エージェントファースト」のIDE

AntigravityはVS Code（またはWindsurf）のフォークをベースとした**IDE型**のツールだ。2つのビューが特徴的で、特に**Manager View**は他のツールにない概念だ。

- **Editor View**：通常のIDEとして使える。インライン補完やコマンド入力もできる
- **Manager View**：複数のエージェントを並行して起動し、**バックグラウンドで非同期に**タスクを実行させる

エージェントが作業した結果は「**Artifacts**」として提供される。スクリーンショット、タスクリスト、実装計画など、ログをスクロールするのではなく、成果物として確認できる。

料金：個人は現在プレビュー無料。Gemini 3.1 Proに寛大なレート制限付き。Claude Sonnet 4.6やGPT-OSSも利用可能。

## 機能比較

| 項目 | Claude Code | Google Antigravity |
|------|-------------|-------------------|
| インターフェース | CLI（ターミナル） | IDE（GUI） |
| 主要モデル | Claude Sonnet/Opus 4.6 | Gemini 3.1 Pro（他モデルも可） |
| 複数エージェント並行 | ❌（単一セッション） | ✅ Manager View |
| ブラウザ操作 | ❌ | ✅ |
| SSHリモート接続 | ✅（そのまま使える） | △（Remote SSH拡張経由） |
| エディタ非依存 | ✅ | ❌（専用IDE必要） |
| 料金 | Pro $20/月〜（OAuth）またはAPI従量課金 | 個人無料（プレビュー） |
| セキュリティ制御 | ❌（許可/拒否設定なし） | ✅ Allow/Deny List |
| CI/CD統合 | ✅ | △ |

## インフラ・運用エンジニアが使う場面で比較する

### ① IaC（Terraform / Ansible）の作業

**Claude Code が向いている**

Terraformファイルの修正、plan結果の解析、エラーの特定と修正——これらはすべてターミナル上で完結する作業だ。Claude Codeはそのまま`terraform plan`を実行し、出力を読み取って修正までを一気通貫でこなせる。

```bash
# 実際の使い方イメージ
claude "modules/network/main.tf のサブネット設定を見て、
        dev/prod/stg の3環境分を自動生成して"
```

エディタを開かずSSHセッション上でそのまま動かせるのは、リモートサーバーでの作業が多いインフラエンジニアには大きなメリットだ。

**Antigravity は**ローカル環境でのIaC開発には使えるが、SSHリモートでの作業や、CI/CDパイプラインへの組み込みには一手間かかる。

---

### ② 監視設定・アラートルール

**Antigravity が向いている**

PrometheusのルールファイルやGrafanaダッシュボードのJSON編集など、**設定ファイルと実際の画面を行き来する作業**にAntigravityのブラウザ操作が活きる。

「Grafanaでこのグラフが出るようにアラートルールを作って、ブラウザで確認して」という指示を、エージェントが自律的にこなせる。これはClaude Codeではできない。

---

### ③ インシデント対応・ログ調査

**Claude Code が向いている**

障害時の対応は「今すぐ、ターミナルで」が基本だ。Claude Codeはすでに開いているターミナルセッションにそのまま入り込める。

```bash
# ログを読ませながら原因特定
cat /var/log/nginx/error.log | claude "このエラーの原因と対処法を教えて"
```

Antigravityを起動してIDEを開いて、という手順はインシデント中には現実的でない。

---

### ④ CI/CDパイプライン、スクリプト自動化

**Claude Code が向いている**

GitHub ActionsのYAMLやJenkinsfileの生成・修正、シェルスクリプトのデバッグはCLIとの親和性が高い。Claude Codeはパイプラインのステップ内から直接呼び出すことも可能だ。

---

### ⑤ 長時間かかる作業のバックグラウンド実行

**Antigravity が向いている**

「このリポジトリ全体のTerraformをモジュール化して」「Kubernetes YAMLを全部Helmチャートに変換して」——こういった大規模なリファクタリングタスクをAntigravityのManager Viewに投げ、別の作業をしながら待つことができる。

Artifactsで作業内容を後から確認できるため、「何をやったのか」が追いやすい。

---

## 使い分けの結論

```
インシデント対応、リモートSSH作業、CI/CD統合  →  Claude Code
ローカル開発、ブラウザ確認が必要な作業、大規模リファクタリング  →  Antigravity
```

インフラ・運用エンジニアの**日常的な作業の7割はCLIで完結する**。SSH接続先のサーバーで動かせる、エディタ不問で使える、という点でClaude Codeのほうが刺さる場面が多い。

一方、**ローカル環境でのインフラコード開発**や、**画面確認を伴う検証作業**、**週次の大規模タスクをバックグラウンド実行**するような場面ではAntigravityの設計思想が活きる。

両者は競合というより**補完関係**に近い。実際、Antigravityはモデルとして Claude Sonnet 4.6をサポートしており、インフラ系エンジニアが両方使い分けるという運用も現実的だ。

## まとめ

| シナリオ | 推奨 |
|---------|------|
| SSH接続先での作業 | Claude Code |
| ターミナルが主戦場の日常業務 | Claude Code |
| CI/CDパイプライン組み込み | Claude Code |
| ローカルでのIaC開発 | どちらでも |
| 監視ツールのUI確認を伴う設定 | Antigravity |
| 大規模リファクタリングの非同期実行 | Antigravity |
| コスト重視・まず試したい | Antigravity（無料） |

**最初に試すなら**：Antigravityは個人プレビューが無料なので、まず触ってみる価値はある。ただし日常業務でCLIとターミナルが中心なら、Claude Codeの「エディタを開かなくていい」体験は思った以上に快適だ。
